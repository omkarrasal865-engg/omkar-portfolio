Design a vending machine for blind people.

Framework: 3 stages of interaction (Discovery → Operation → Retrieval) → Senses → Edge cases

↓
Stage 1 — Discovery

Bluetooth beacon — "Vending machine 8 meters ahead, slightly left" · Soft distinct hum on approach within 2m · Tactile ground path — raised floor strips leading to machine.

Stage 2 — Operation

Braille labels on every button · Headphone jack for private audio · Voice interface modes — category mode, budget mode, quick mode ("same as last time") · Each item described — name, price, ingredients, allergens.

Stage 3 — Retrieval

"Your item is ready. Collect from the bottom tray." · Raised edges — product can't fall off · 15-second window with audio reminders.

Edge Cases

Out of stock → "Similar option at ₹35?" · Payment failed → "Use cash slot on your right" · No smartphone → machine fully operable via tactile keypad alone.

Key Insight: The app enhances but the machine must be fully operable without a phone. The app is the premium layer — not the only layer. Hardware PM rule: never make the core use case device-dependent.

Root Cause Analysis✓ Solved

Uninstalls of a mobile banking app are increasing. Identify the root cause.

Framework: Scope → Internal (update? campaign? bugs?) → External (competitors? regulation?) → Root Cause

↓
Context

15-year-old legacy private bank in India going digital. Uninstalls started rising gradually last weekend. Both Android and iOS affected. A UI update was released 8–9 days ago. A marketing campaign offering credit to new installers was also recently active.

Investigation Path

Internal: Recent UI update (8–9 days ago, matches timing) · Marketing campaign for new installers · No significant bugs found · No feature additions or deletions · External: No data leaks, no PR issues, no regulations · New neo-bank targeting students with promotional features.

Root Cause

The marketing campaign acquired users with low product stickiness — users came for the free credit, utilized it, then had no compelling reason to stay. The neo-bank was NOT the root cause — loyal banking consumers don't switch that drastically in a week.

Recommendations

Segment users into cohorts by usage patterns · For campaign-acquired users with low activation: improve onboarding and activation touchpoints · For acquired-and-activated users who still uninstall: investigate specific feature drop-offs · Track campaign-driven vs. organic churn separately.

Key Insight: Always investigate campaign timing vs. metric change timeline as an early hypothesis in any uninstall/churn RCA. Promotional installs create artificial spikes that collapse — they mask real retention problems.

A key metric for a video streaming service dropped 80%. What do you do?

Framework: Clarify → Data Integrity → Internal/External → Segment → Correlate → Act

↓
First Move — Clarify

Ask: "Which metric dropped?" Never jump to solutions. Which metric (DAU, watch time, revenue)? 80% vs what baseline (yesterday, last week)? All users or a specific segment? Sudden or gradual? What changed around that time?

Universal Diagnostic Framework
Step 1Is it a data/tracking bug? (80% is suspicious — verify the data is real first)
Step 2Is it internal? Recent deployment, feature flag, infrastructure outage?
Step 3Is it external? Competitor launched, app store issue, seasonal?
Step 4Is it a segment issue? Specific geography, device, cohort?
Step 5Correlate with last change — deployment timestamp, campaign, content removal
Immediate Actions

First 30 mins: Check analytics tracking, system dashboard, alert engineering. First 2 hours: Segment the drop, check deployment correlation. First 24 hours: Root cause identified, hotfix deployed, stakeholder comms sent.

Key Insight: An 80% drop almost always means a data tracking bug or a recent deployment broke something. Diagnose before prescribing. Spending on fixes before you know the root cause wastes time and trust.

Product Metrics✓ Solved

What metrics would you use to measure the success of the Save feature at Facebook?

Framework: North Star → Primary → Secondary → Guardrail Metrics

↓
Core Insight

Saving a post is only half the job. The real value is coming back to it later. A save never revisited is a failed save experience — like bookmarking 100 tabs you never open.

North Star Metric

Saved posts revisited within 7 days / Total saves — captures the complete save-retrieve loop, not just the save action.

Metric Hierarchy

Adoption: % of DAU using Save · saves per active user per week · Engagement: Save-to-revisit rate · time between save and revisit · saves by content category · Retention: Saved collection return rate · 30-day feature retention.

Guardrails

Overall newsfeed engagement · Time spent on Facebook · Content creator reach — saving should not reduce any of these.

Key Insight: Users who view the same content 2–3x are prime candidates for a Save prompt. Behavior → product trigger. That behavioral proxy is a better signal than asking users if they want to save.

How would you measure the success of Facebook Likes?

Framework: Three-party (Liker + Creator + Platform) → Like Quality → Guardrails

↓
Three-Party Framework

Liker — expression value: like rate per session, time on post before liking · Creator — validation value: likes per post, posting frequency post-likes · Platform — algorithm signal: like accuracy rate, like-bait detection.

North Star Metric

Like-to-meaningful-engagement conversion rate — % of liked posts where the user also commented, shared, or saved.

Guardrails

Over-optimization of free-to-paid conversion can cause short-term sign-ups that churn quickly — inflating costs without sustainable revenue. Conversion rate is a leading indicator; retention is the lagging confirmation.

Product Metrics✓ Solved

Identify one metric for Facebook Marketplace that the whole team should monitor daily.

Framework: The "one metric" test — captures ALL stakeholder value simultaneously

↓
The One Metric

Transaction Completion Rate — % of buyer-seller conversations that result in a completed transaction.

Why This Above All Others

High listings + low transaction rate = marketplace full of unsold items → sellers leave → marketplace dies. This metric can only be high if supply quality AND buyer experience AND trust AND pricing all work. It captures both sides simultaneously.

What a Drop Diagnoses

Listing quality declining → seller onboarding · Buyers not trusting → verification · Search broken → algorithm · Payment friction → UX.

Guardrail

Fraud rate — transaction completion must rise WITHOUT fraud rising. Otherwise you're just completing bad transactions.

Key Insight: For "pick ONE metric" questions — find the metric whose drop immediately signals something is broken AND whose improvement automatically improves everything else. That's the north star test.
